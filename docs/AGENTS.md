# AGENTS.md

이 문서는 SoundCluster에서 AI 코딩 에이전트가 프로젝트 맥락을 빠르게 이해하고, 기존 아키텍처와 개발 규칙을 일관되게 지키며 작업하도록 돕는 기준 문서입니다. Codex를 포함한 모든 에이전트는 이 문서를 프로젝트 가드레일로 사용합니다.

---

## 1. 프로젝트 개요

- **프로젝트명**: SoundCluster
- **목표**: 사용자가 검색한 음악의 가사와 메타데이터를 기반으로 5차원 감성 벡터를 분석하고, 이를 3D 공간 좌표로 변환해 React Three Fiber 기반의 음악 성단으로 시각화합니다.
- **핵심 흐름**: 음악 검색 -> 백엔드 분석 요청 -> Gemini JSON 분석 -> MySQL 캐싱 -> SSE 스트리밍 -> 클라이언트 상태 반영 -> MDS 좌표 변환 -> 3D 성단 렌더링

---

## 2. 기술 스택

### Frontend

- React + TypeScript + Vite
- React Three Fiber, Three.js
- CSS Modules
- React Hooks, Context API

### Backend

- Node.js + Express
- Server-Sent Events
- MySQL
- `@google/genai`

### Shared

- 공통 타입, 상수, 순수 유틸리티는 `shared/` 아래에서 관리합니다.
- 클라이언트와 서버가 함께 참조하는 데이터 구조는 `shared/types/`에 먼저 정의합니다.

---

## 3. 작업 원칙

### 먼저 파악하고 작게 변경

- 작업 전 `docs/Architecture.md`, `docs/checklist.md`, 관련 소스 파일을 먼저 확인합니다.
- 변경 범위는 요청된 기능과 직접 관련된 파일로 제한합니다.
- 대규모 구조 변경, 새 디렉토리 추가, 의존성 추가가 필요하면 이유와 영향 범위를 먼저 설명합니다.

### 기존 구조 우선

- 파일 위치는 `docs/Architecture.md`의 레이어 구분을 따릅니다.
- 이미 존재하는 패턴, 네이밍, 모듈 경계를 우선합니다.
- 공통화가 필요한 값은 `shared/constants/` 또는 각 레이어의 `constants` 파일로 이동합니다.

### 검증 가능한 단위로 구현

- 기능은 작은 단계로 나누어 구현합니다.
- 각 단계마다 실행 가능한 검증 방법을 남깁니다.
- 체크리스트 항목을 완료한 경우 `docs/checklist.md`의 해당 항목을 갱신합니다.

## 4. 코딩 규칙

### TypeScript

- `strict` 기준을 지킵니다.
- `any`는 사용하지 않습니다. 알 수 없는 외부 입력은 `unknown`으로 받은 뒤 검증합니다.
- API 응답, 감성 벡터, Spotify 메타데이터, SSE 이벤트는 명시적인 interface 또는 type으로 정의합니다.

### React

- 컴포넌트 파일은 PascalCase를 사용합니다.
- 상태 공유는 React Hooks와 Context API를 사용합니다.
- Redux, Zustand 등 별도 상태 관리 라이브러리는 도입하지 않습니다.
- 비즈니스 로직, 데이터 가공, 계산 로직은 컴포넌트에서 분리합니다.

### Styling

- CSS Modules만 사용합니다.
- Tailwind CSS와 외부 UI 컴포넌트 라이브러리는 사용하지 않습니다.
- 색상, 간격, 타이포그래피는 토큰화된 값으로 관리합니다.
- 컴포넌트 내부에 임의의 하드코딩 색상과 매직 넘버를 늘리지 않습니다.

### Three.js / R3F

- `useFrame` 내부에서는 매 프레임 객체 생성을 피합니다.
- `THREE.Vector3` 등 반복 사용 객체는 `useRef`나 외부 스코프에서 재사용합니다.
- 이동 보간은 `delta`를 반영해 주사율 차이에 따른 애니메이션 편차를 줄입니다.
- 성능이 중요한 렌더 루프에서는 제한적으로 mutable 패턴을 허용합니다.

### Backend

- API Key와 시크릿은 프론트엔드에 노출하지 않습니다.
- 환경 변수는 서버의 `.env`에서만 읽습니다.
- Gemini 응답은 반드시 JSON 파싱과 스키마 검증을 거친 뒤 클라이언트로 전달합니다.
- SSE 이벤트 타입과 payload 구조는 공유 타입 또는 명확한 서버 타입으로 관리합니다.

## 5. 코드 품질 기준

- 변수명과 함수명으로 의도를 드러냅니다.
- 설명용 주석은 남발하지 않습니다. 복잡한 알고리즘, 외부 제약, 성능상 의도가 있는 경우에만 짧게 남깁니다.
- 과도한 체이닝이나 중첩 삼항 연산자를 피합니다.
- 수학 계산은 중간 값을 명확한 이름의 변수로 분리합니다.
- 데이터 가공은 가능한 한 순수 함수와 불변 데이터를 사용합니다.
- 렌더 루프나 대량 계산처럼 성능이 중요한 구간에서는 목적이 분명한 가변 처리를 허용합니다.

## 6. 네이밍 및 파일 규칙

- 컴포넌트: `PascalCase.tsx`
- 훅: `useSomething.ts`
- 일반 함수와 변수: `camelCase`
- 타입과 인터페이스: `PascalCase`
- CSS 클래스: `kebab-case`
- 문서 파일명은 현재 레포의 실제 대소문자를 기준으로 참조합니다.
  - `docs/AGENTS.md`
  - `docs/Architecture.md`
  - `docs/checklist.md`
  - `docs/prompt-guideline.md`

## 7. 에이전트 작업 절차

1. 요청과 관련된 문서 및 소스 파일을 확인합니다.
2. 기존 구조와 체크리스트에서 해당 작업의 위치를 찾습니다.
3. 필요한 변경을 최소 범위로 구현합니다.
4. 포맷, 타입 검사, 테스트, 실행 확인 중 가능한 검증을 수행합니다.
5. 변경 파일과 검증 결과를 간결하게 보고합니다.

불확실한 요구사항이 있어도 합리적인 기본값이 명확하면 진행합니다. 단, 사용자 경험, 데이터 모델, 디렉토리 구조, 외부 의존성, 보안 정책에 영향을 주는 결정은 먼저 확인합니다.

## 8. 커밋 메시지 규칙

커밋을 요청받은 경우 아래 형식을 사용합니다.

```text
<type>: #<issue-number> <summary>

- 확인내용: <구현 또는 수정한 핵심 내용>
- 이해 안 됐던 부분: <작업 중 확인이 필요했거나 해결 과정에서 판단한 내용>
```

이슈 번호가 없는 작업은 사용자에게 확인하거나, 레포의 기존 커밋 관례가 있다면 그 관례를 우선합니다.

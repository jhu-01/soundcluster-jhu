# 🤖 GEMINI.md (AI Agent System Instruction)

본 문서는 SoundCluster 프로젝트 빌드업 과정에서 AI 에이전트(Gemini)에게 명확한 페르소나, 개발 컨벤션, 기술 제약 조건 및 프롬프트 스킬을 주입하여, 아키텍처 오염을 방지하고 정형화된 데이터 파이프라인을 사수하도록 하는 마스터 지침서입니다.

---

## 1. 프로젝트 개요
- **프로젝트 명**: SoundCluster
- **설명**: 사용자가 검색한 음악의 가사와 메타데이터를 기반으로 AI(Gemini SDK)가 N차원 감성 수치를 정밀 분석하고, 이를 차원 축소(MDS) 알고리즘을 거쳐 3D 웹 우주 공간(R3F/WebGL)에 나만의 음악 성단으로 실시간 시각화 및 공유할 수 있는 몰입형 감성 음악 대시보드 서비스.

---

## 2. 기술 스택 및 제약 사항
Gemini 에이전트는 코드 생성 및 아키텍처 설계 시 아래의 스택과 제약 사항을 **반드시** 준수해야 합니다.

### [Architecture & Core]
- **Frontend**: React + TypeScript (Vite 환경) + React Three Fiber (R3F) / Three.js
- **Backend**: Express.js (Node.js) + SSE (Server-Sent Events) 실시간 데이터 푸시 스트리밍
- **LLM Integration**: 구글 공식 `@google/genai` SDK 활용 (application/json 구조화 강제)
- **Data Storage**: MySQL 데이터베이스 활용 (분석된 5차원 원본 데이터를 `tracks` 및 `shared_spaces` 테이블에 영구 적재 및 캐싱)

### [Strict Coding Rules]
- **Styling**: **CSS Modules 전용** (외부 UI 컴포넌트 라이브러리 및 Tailwind CSS 사용 엄금)
- **3D Optimization**: 
  - `useFrame` 애니메이션 루프 내에서 오브젝트 이동 시 **매 프레임 선형 보간(lerp)** 연산을 사용하되, 기기별 주사율 편차 방지를 위해 반드시 `delta` 타임을 곱하여 보정할 것.
  - 매 프레임 내부에서 `new THREE.Vector3()` 등 가비지 컬렉션을 유발하는 인스턴스 반복 생성 금지, 상단 `useRef` 격리 후 메모리 재사용(Mutation) 필수.
- **Clean Code & Modularity**:
  - **하드코딩 금지 (상수화)**: API 엔드포인트, Base URL, SSE 스트리밍 주소, 고정된 환경 설정 값 등 반복되거나 변경 가능성이 있는 문자열/수치는 절대 함수나 컴포넌트 내부에 하드코딩하지 않는다. 반드시 별도의 `constants` 파일(예: `config.ts`, `apiEndpoints.ts`)로 추출하여 `const` 혹은 `readonly object`로 관리해야 한다.
  - **관심사 분리 (함수 추상화)**: 특정 컴포넌트나 라우터 내부에 종속되지 않는 독립적인 비즈니스 로직, 데이터 가공(포매팅, 행렬 연산, MDS 파싱 등), 유효성 검사 로직은 반드시 별도의 순수 함수(Pure Function)로 분리하여 `utils` 폴더 내에 모듈화하고 재사용성을 극대화한다.
- **State Management**: **React Hooks & Context API 내장 기능만 사용** (Redux, Zustand 등 외부 라이브러리 금지)
- **Type System**: Strict TypeScript (`any` 타입 사용 절대 금지, 명확한 Interface 정의 필수)
- **Security**: Spotify API 및 Gemini API Key는 절대로 프론트엔드에 노출하지 않으며, 오직 Express의 `.env`를 통해서만 관리하고 백엔드 서버에서 호출을 대행함.

## 3. 개발 컨벤션 (Convention)
- **Components**: PascalCase (예: `SearchBar.tsx`, `ControlPanel.tsx`, `StarNode.tsx`)
- **Naming**: 변수/함수는 `camelCase`, CSS 클래스명은 `kebab-case` 준수
- **Design Spec**: 사전에 정의된 네온 그리드 사양, 컬러 토큰, 타이포그래피 사양을 CSS Modules 내에서 일관되게 준수
- **Git Commit**: 단순한 기능 요약이 아닌 **회고형 커밋(Reflective Commit)** 구조를 준수한다.
  - **제목**: `<type>: #<이슈번호> <기능명>` (예: `feat: #3 별 노드 생성 및 매 프레임 lerp 감속 애니메이션 안정화`)
  - **본문**: 아래 두 가지 키워드를 불릿 포인트(`-`)로 반드시 포함하여 작성한다.
    1. `- 확인내용:` (구현된 핵심 로직, 검증된 기능, 수정된 버그 상세 나열)
    2. `- 이해 안 됐던 부분:` (작업 중 발생한 에러, 복잡했던 상태 동기화 로직, 트러블슈팅 과정)

## 4. 에이전트 행동 수칙 (Agent Behavior Rules)
Gemini는 사용자와 대화하거나 코드를 생성할 때 다음 4가지 수칙을 엄격하게 이행해야 합니다.

1. **[구현 전략 선행]** 코드를 작성하기 전, 구현하려는 기능의 **전략 및 아키텍처(프론트-백엔드 간 데이터 흐름, 컴포넌트 구조)**를 사용자에게 먼저 명확히 설명하고 합의를 구한다.
2. **[작은 단위 분할]** 모든 코드는 한 번에 거대하게 작성하지 않고, 사용자가 **리뷰 및 단계별 테스트를 진행하기 좋은 작은 단위**로 끊어서 제공한다.
3. **[임의 결정 금지]** 기획서에 명시되지 않은 UI/UX 사양, 기능 스펙 또는 예외 처리 방식은 독단적으로 결정하지 말고, **반드시 사용자에게 먼저 질문하여 확인**한다.
4. **[디렉토리 생성 승인]** 새로운 디렉토리나 구조적 파일 변경이 필요할 경우, 실제 작업을 수행하기 전에 **사용자에게 파일 트리를 먼저 보고하고 승인을 받는다.**

## 🛠️ 5. 맞춤형 프롬프트 스킬 (AI Skills)

### 🌟 SKILL 1: Commit-Generator
[Skill: Commit-Generator]
작업 내용: <예: 방금 작업한 코드의 주요 내용, 발생했던 에러, 해결 방법 등>
지침:
1. 제공된 작업 내용을 바탕으로 컨벤션 규칙에 맞는 커밋 메시지를 작성해줘.
2. 제목은 `<type>: #<이슈번호> <제목>` 포맷으로 작성해.
3. 본문은 반드시 `- 확인내용:`과 `- 이해 안 됐던 부분:` 두 가지 섹션으로 나누어 작성해.
4. 특히 '이해 안 됐던 부분'에는 작업 중 겪었던 복잡한 로직(예: useEffect 상태 동기화)이나 에러 해결 과정을 개발자의 시선에서 구체적으로 기록해줘.

### 🌟 SKILL 2: Checklist-Driven-Build
[Skill: Checklist-Driven-Build]
목표: checklist.md의 태스크를 기준으로 삼아, 에이전트가 구현 범위(Scope)를 벗어나거나 프로젝트 컨벤션을 위반하는 코드를 짜지 않도록 자율 제어 및 검증합니다.

지침 및 단계별 수행 규칙:
에이전트는 사용자가 특정 기능 구현을 요청하거나, 다음 체크리스트 항목을 진행하라고 지시할 때 반드시 아래 3단계 '자율 검증 워크플로우'를 수행해야 합니다.

1단계. [구현 전 계획 및 체크리스트 매핑]
- 작업을 시작하기 전, `checklist.md`를 먼저 읽고 현재 구현하려는 기능이 어떤 마일스톤과 태스크에 해당하는지 명확히 식별하십시오.
- `GEMINI.md` 수칙 1번에 따라, 곧바로 코드를 생성하지 말고 해당 태스크를 해결하기 위한 React-Express 간의 구조적 구현 전략을 사용자에게 먼저 제안하고 확인받으십시오.

2단계. [구현 중 디렉토리 및 파일 위치 실시간 감시]
- 코드를 작성하거나 파일을 생성할 때, 해당 파일의 위치가 `architecture.md`에 명시된 폴더 구조와 완벽히 일치하는지 대조하십시오.
- 만약 구현 과정에 필요한 상위/하위 디렉토리가 누락되었거나 부족함을 감지하면, 임의로 폴더를 만들거나 다른 곳에 파일을 생성하지 마십시오. 즉시 해당 단계에서 구현을 중단(Undo/되돌리기)하고, 사용자에게 상황을 보고한 뒤 필요한 디렉토리 구조 승인을 먼저 받으십시오.

3단계. [구현 후 CSS 디자인 토큰 및 품질 검증]
- 특히 스타일링(CSS Modules) 작업이 포함된 경우, `design.md`에 정의된 컬러 토큰 및 타이포그래피 변수(--color-*, --font-*)가 `tokens.css`나 전역 스타일에 맞게 올바르게 바인딩되었는지 코드를 한 줄씩 재검토하십시오.
- 만약 하드코딩된 색상 값(예: #000, #FFFFFF)이나 규격에 맞지 않는 스타일 속성이 발견되면 코드를 즉시 재수정(Refactor)하여 디자인 토큰을 강제 적용하십시오.

위 3단계 검증이 모두 성공적으로 통과하면, `checklist.md`에서 완료된 해당 Task 항목을 체크([x]) 상태로 업데이트한 뒤, 수정된 파일 트리와 함께 작업 완료 보고를 사용자에게 제출하십시오.
